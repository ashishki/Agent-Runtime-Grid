# Failure Modes

Agent Runtime Grid treats Redis Streams as a delivery mechanism and Postgres as the authoritative lifecycle record. Failure handling must preserve the job event trail and must never create duplicate terminal finalizations.

## Worker Crash After Lease

Scenario:

1. A worker leases a Redis Streams entry through the consumer group.
2. The worker records the job as `running`.
3. The worker exits before acknowledging the stream entry or writing a terminal state.
4. The pending stream entry becomes stale after the configured lease timeout.
5. Recovery inspects Redis pending entries, loads the matching job under a database lock, and decides from Postgres state.

Expected behavior:

- If the job is already terminal in Postgres, recovery acknowledges the stale stream entry and does not change lifecycle state.
- If the job is non-terminal and retry budget remains, recovery records `stale_lease_recovered`, requeues the job with the next attempt number, and acknowledges the stale entry.
- If the job is non-terminal and retry budget is exhausted, recovery records terminal `failed` with `StaleLeaseExhaustedError`, moves the stale message to the Redis DLQ, and relies on idempotent finalization to prevent duplicate terminal events.

Proof:

- `tests/integration/test_stale_lease_recovery.py::test_stale_lease_is_detected`
- `tests/integration/test_stale_lease_recovery.py::test_stale_job_requeues_and_completes_once`
- `tests/integration/test_stale_lease_recovery.py::test_exhausted_stale_recovery_routes_to_dlq_without_duplicate_finalization`

## Transient Job Failure

Scenario:

1. A worker executes a job and receives a retryable transient error.
2. The job attempt number is still within `max_retries`.

Expected behavior:

- The state machine records `retry_scheduled`.
- A new queue message is published with the next attempt number.
- The original Redis entry is acknowledged.
- If retry budget is exhausted, the job is finalized as `failed`.

Proof:

- `tests/integration/test_worker_lifecycle.py::test_transient_error_requeues_until_retry_limit`
- `tests/load/test_reliability_proof.py::test_v1_proof_runs_against_runtime_state`

## Timeout

Scenario:

1. A job exceeds its configured timeout.

Expected behavior:

- The worker records terminal `timed_out`.
- The Redis entry is acknowledged.
- The terminal transition is guarded by the finalization table.

Proof:

- `tests/integration/test_timeout_cancellation.py::test_worker_times_out_slow_job`
- `tests/load/test_reliability_proof.py::test_v1_report_contains_required_runtime_evidence`

## Cancellation

Scenario:

1. A job is cancelled before execution or while running.

Expected behavior:

- Queued cancellation records terminal `cancelled` without worker execution.
- Running cancellation stops the worker-side execution path and records terminal `cancelled`.
- Acknowledgement happens only after lifecycle state is recorded.

Proof:

- `tests/integration/test_timeout_cancellation.py::test_queued_job_can_be_cancelled_before_worker_execution`
- `tests/integration/test_timeout_cancellation.py::test_running_job_can_be_cancelled`

## Duplicate Delivery Or Submission

Scenario:

1. Redis delivers a replayed message, or a caller repeats the same idempotency key.

Expected behavior:

- Matching idempotency replays return the existing job.
- Mismatched duplicate payloads are rejected.
- Replayed queue messages for terminal jobs are acknowledged as no-ops.
- Terminal finalization is guarded by a database uniqueness constraint.

Proof:

- `tests/integration/test_job_repository.py::test_duplicate_idempotency_key_returns_existing_job`
- `tests/integration/test_idempotent_finalization.py::test_replayed_message_after_finalization_is_noop`
- `tests/integration/test_idempotent_finalization.py::test_racing_workers_produce_one_terminal_event`

## Failure Report Pack

Operator-readable reports are generated by:

```bash
python -m agent_runtime_grid.cli failure-reports write-pack \
  --output-dir reports/failure-injection
```

Generated reports:

- `transient-retry.md`
- `timeout.md`
- `cancellation.md`
- `stale-worker.md`
- `duplicate-finalization.md`
- `dlq.md`

Each report includes scenario, command, expected behavior, actual lifecycle, event trail, metrics, artifact evidence, and known limits.
